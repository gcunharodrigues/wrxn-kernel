# Security review — gate-06 (recon-wrxn gate runbook)

- **Slice:** `gate-06` — bring the published sibling `recon-wrxn` under the `wrxn-main-gate` model.
- **Artifact reviewed:** build commit `3896337` → `acceptance/gate-redesign/recon-wrxn-runbook.md` (docs-only slice; the runbook drives a one-time operator change to a PUBLISHED package's CI/CD + branch protection).
- **Context:** issue `06-apply-gate-recon-wrxn.md`, `PRD.md`, ADR `0007`, `carry-forward.md`.
- **Method:** read the runbook diff; traced every claim to ground truth in the real repo `/_projects/recon-wrxn` (read-only) and the kernel templates it adapts (`lib/protect.cjs`, `.github/workflows/release.yml`, `test/protect.test.cjs`); ran `wrxn ci` read-only against recon to verify the hard-fail claim. No source/test/workflow files were mutated; this report is the only write.

## Verdict: PASS-WITH-FINDINGS

The runbook is sound and carries every prior hardening. The apply order avoids a lockout and a protection gap, OIDC tokenless+provenance is kept, the slice-05 MED-1 token-scoping hardening is faithfully reproduced, double-publish is prevented (and backstopped by npm version-immutability + the `npm view` guard), and the own-suite-CI choice is **not** a security downgrade — it is verified-correct and avoids a self-inflicted permanent-lockout. Findings are LOW/INFO operational-completeness notes; none block. The single actionable one (SEC-LOW-1) is that removing the NPM_TOKEN *job* does not remove/revoke the NPM_TOKEN *secret*, so the long-lived credential the OIDC model is meant to retire survives until the operator deletes+revokes it.

## Posture calls (the four requested)

### 1. Apply-order / no-lockout / no-gap — PASS

- **No permanent PR lockout.** The runbook orders: land the workflows → open a throwaway PR so `wrxn-ci` reports once → THEN apply the ruleset (runbook "Apply order" + Steps 1–4). A required check that never reported leaves PRs pending forever; landing the workflow first prevents it. Reinforced by two correct, load-bearing details: the check job is named **`wrxn-ci`** (matches the ruleset's required context `wrxn-ci`, `lib/protect.cjs:60`), and a **single fixed node version** is used on purpose — a matrix would emit `wrxn-ci (20)`/`wrxn-ci (22)`, neither matching the required `wrxn-ci` → un-satisfiable → lockout. The runbook explicitly avoids this.
- **`bypass_actors: []` (no bypass) — same as the kernel.** Verified in `lib/protect.cjs:41`; the SAME `buildRulesetSpec` protects recon (no recon-specific branch). The walk-evidence checklist asserts `bypass_actors: []` (runbook "Walk evidence"). `test/protect.test.cjs:63,198-211` proves the recon slug gets `~DEFAULT_BRANCH` + empty bypass with a fake invoker (no real `gh`).
- **Unprotected window is inherent + accepted.** Between landing workflows and applying the ruleset, recon `main` is unprotected (runbook is honest about this). This is unavoidable in any land-then-apply bootstrap (you cannot require a check that has never run) and matches the kernel self-host. Operator-controlled and brief → INFO (SEC-INFO-1), not a defect.
- **Residual operational dependency (SEC-LOW-2):** `wrxn protect` is fail-SOFT — on no-`gh`/no-admin it prints a skip and **exits 0**, leaving recon unprotected. The no-gap guarantee therefore depends on the operator completing the walk-evidence checklist (esp. "direct push to `main` is blocked"). The runbook already mandates that evidence, so this is a flagged dependency, not a hole.

### 2. Secret hygiene — PASS (with SEC-LOW-1)

- **OIDC tokenless + provenance KEPT.** The adapted `release.yml` uses `id-token: write` + `npm publish --provenance` + no `NPM_TOKEN`; verified recon already has trusted-publishing (`package.json publishConfig.provenance:true`, existing `release.yml:32`).
- **Legacy long-lived NPM_TOKEN publish job — REMOVAL is instructed.** Confirmed the job exists: `recon-wrxn/.github/workflows/ci.yml:36-64` (`publish:`, `if: startsWith(github.ref,'refs/tags/v')`, `NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}`). The runbook's "Reconcile" step correctly orders its removal.
- **slice-05 hardening carried verbatim.** The adapted `release.yml` reproduces gate-05 MED-1: `persist-credentials: false` on checkout + an isolated, env-scoped `x-access-token` final tag-push step, so the npm `ci`/`build`/`test`/`publish` steps (third-party lifecycle code) hold **no** ambient `contents:write` token. Matches kernel `.github/workflows/release.yml:45-101` line-for-line (plus the recon-correct additions: `npm run build`, `--tag latest`, `npx --yes @gcunharodrigues/wrxn release-check`).
- **No secret reaches PR-from-fork code.** `release.yml` triggers on `push: branches:[main]` only (never `pull_request`), so the OIDC publish identity is unreachable until AFTER merge (post review + ruleset). `wrxn-ci.yml` is `pull_request` with `permissions: contents: read` and references **no** secrets. The kept `recon-review.yml` uses `pull_request` (not the dangerous `pull_request_target`) → fork PRs get a read-only `GITHUB_TOKEN` and inject the report via `env:`/`process.env` (not inline `${{ }}`), so no token escalation or script injection to fork code. No `pull_request_target` exists anywhere in recon's workflows (all three read).

### 3. CD double-fire / integrity — PASS

- **No double-publish.** New `release.yml` triggers on `push: branches:[main]` and pushes a `v<version>` tag; that tag does NOT re-trigger it (branches-only). The runbook removes BOTH legacy tag-`v*` publish paths (ci.yml `publish` job + old tag-triggered `release.yml`), so the pushed tag fires no second publisher. Triple-backstopped: (a) the retained `npm view "$PKG@$VER"` idempotency guard skips an already-published version; (b) npm version-immutability rejects any republish (409) even if the operator forgets a removal; so double-PUBLISH is effectively impossible. The NPM_TOKEN-job removal is therefore primarily a *secret-hygiene* act (see SEC-LOW-1), not a double-publish risk.
- **Prerelease dist-tag handled correctly — not a wrong clobber.** `npm publish --provenance --access public --tag latest`. recon's version `6.0.0-wrxn.6` is a semver prerelease and the `-wrxn.N` line IS recon's shipping line — its EXISTING `release.yml:32` already publishes `--tag latest`. The runbook **preserves** recon's intentional, current distribution model; it introduces no new `latest` clobber. INFO only (SEC-INFO-2): the hard-coded `--tag latest` should be revisited IF recon ever ships a separate stable line.

### 4. Own-suite CI vs managed-integrity — CONFIRMED, not a downgrade

Reasoned from the protected asset. `managedIntegrity` protects kernel-MANAGED files; recon-wrxn has **no `wrxn.install.json` and no managed payload**, so there is nothing for it to verify — it is **vacuous, not bypassed**. Verified live (read-only `node bin/wrxn.cjs ci --root …/recon-wrxn` → **exit 2**): `✓ managed-integrity — not a wrxn install, nothing to verify`, `✓ wiki-lint`, `✗ synapse-manifest — no manifest` (the hard-fail), `✓ json-validity`, `✓ node-check`. Two conclusions:

1. **Dropping `npx wrxn ci` removes no protection** — the only thing it would "verify" (managed integrity) does not exist on recon; the universal step exists to give a *no-suite* install a backstop, the exact opposite of recon.
2. **Invoking it would be actively harmful** — `synapseManifestLint` hard-fails on the absent manifest, so the required `wrxn-ci` check could never go green → a self-inflicted permanent denial-of-merge. NOT running it is the correct, lockout-avoiding call.

recon instead gets a STRONGER required check (its real `vitest` suite via `npm ci && npm run build && npm test`, verified `package.json test: "vitest run"`) behind the SAME server-side no-bypass ruleset. Server enforcement is uniform; only the check's internal work is appropriately repo-specific. Confirm.

## Injection / traversal / authz / fail-posture sweep (clean)

- **Injection:** the only `${{ }}` reaching a `run:` shell are trusted constants — commit SHAs (`github.event.before`/`github.sha` into `release-check --range`) and the `major|minor|patch` enum (`steps.decide.outputs.bump` echo); no attacker free-text (matches the slice-05 LOW-1 finding, "confirmed safe"). `VER`/`PKG` from `node -p` are gated by the PR+ruleset and quoted; the tag step runs only after a successful publish, and npm semver-validation + version-immutability bound any crafted version string.
- **Path traversal:** the recon slug `gcunharodrigues/recon-wrxn` flows through `parseSlug`'s `SLUG_RE` (`lib/protect.cjs:142-159`) which rejects `..`, spaces, `;`, `$()`, backticks before any value reaches `gh api` (gate-02 LOW-1 hardening).
- **Authz:** ruleset requires a `gh` repo-admin token; `bypass_actors:[]` means even the operator/agent token must go through a PR (break-glass = temporarily disable, per ADR). Intended model.
- **Fail-posture:** CI fails CLOSED (any non-zero npm step → check red → merge blocked); CD fails CLOSED on publish (a failed/absent `decide` skips all publish steps → no publish); `wrxn protect` fails SOFT by design (must not break a remote-less `update`), backstopped by mandatory walk-evidence (SEC-LOW-2).
